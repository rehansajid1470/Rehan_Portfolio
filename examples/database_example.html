Common Database Issues in Python

Working with databases in Python applications often introduces specific challenges and bugs. From SQL injection vulnerabilities to connection management issues, these problems can lead to security breaches, resource leaks, and performance bottlenecks.

Below, I demonstrate how I diagnose and fix the most common database-related bugs in Python applications, ensuring your data remains secure, your code performs optimally, and your application stays stable.

Database Bug Fixes

Bug #1: SQL Injection Vulnerability

One of the most dangerous database bugs is SQL injection vulnerability, where user input is directly incorporated into SQL queries without proper sanitization. This can allow attackers to manipulate the database in unintended ways.

Impact: Security breach, unauthorized data access, data loss, and potential system compromise.

Buggy Code:

import sqlite3

def get_user(username):
    conn = sqlite3.connect('users.db')
    cursor = conn.cursor()
    
    # VULNERABLE: Direct string formatting in SQL query
    query = f"SELECT * FROM users WHERE username = '{username}'"
    cursor.execute(query)
    
    result = cursor.fetchone()
    conn.close()
    return result

# If attacker inputs: admin' --
# The query becomes: SELECT * FROM users WHERE username = 'admin' --'
# This comments out the rest of the query and could bypass authentication

Fixed Code:

import sqlite3

def get_user(username):
    conn = sqlite3.connect('users.db')
    cursor = conn.cursor()
    
    # FIXED: Using parameterized queries
    query = "SELECT * FROM users WHERE username = ?"
    cursor.execute(query, (username,))
    
    result = cursor.fetchone()
    conn.close()
    return result

# Now if someone inputs: admin' --
# It will be treated as a literal string parameter
# The query safely looks for a username with that exact value

Fix Explanation:

The bug was fixed by implementing parameterized queries instead of string formatting. Parameterized queries ensure that user input is treated as data, not executable SQL code, preventing SQL injection attacks. This is a fundamental security practice when working with databases in any language.

Bug #2: Database Connection Leaks

Connection leaks occur when database connections are opened but not properly closed, especially during exceptions. These leaks can exhaust database connection pools and cause performance degradation or application failures.

Impact: Resource exhaustion, performance degradation, and eventual application crashes.

Buggy Code:

import psycopg2

def process_customer_orders(customer_id):
    conn = psycopg2.connect(
        dbname="orders",
        user="app_user",
        password="password",
        host="localhost"
    )
    cursor = conn.cursor()
    
    # If an exception occurs here, the connection is never closed
    cursor.execute("SELECT * FROM orders WHERE customer_id = %s", (customer_id,))
    orders = cursor.fetchall()
    
    for order in orders:
        process_order(order)
        
    conn.close()  # Only reached if no exceptions occur
    return True

Fixed Code:

import psycopg2

def process_customer_orders(customer_id):
    conn = None
    try:
        conn = psycopg2.connect(
            dbname="orders",
            user="app_user",
            password="password",
            host="localhost"
        )
        cursor = conn.cursor()
        
        cursor.execute("SELECT * FROM orders WHERE customer_id = %s", (customer_id,))
        orders = cursor.fetchall()
        
        for order in orders:
            process_order(order)
            
        return True
    except Exception as e:
        print(f"Error processing orders: {e}")
        raise
    finally:
        # Connection is closed even if an exception occurs
        if conn is not None:
            conn.close()

Fix Explanation:

The fix uses a try-finally block to ensure that database connections are always closed, even when exceptions occur. Additionally, a null check is added to handle cases where the connection creation itself might have failed. This pattern is essential for proper resource management in database applications.

Bug #3: Inefficient Query Patterns

Inefficient query patterns, such as N+1 query problems, can significantly degrade application performance. These occur when code executes multiple database queries in a loop instead of using more efficient bulk operations.

Impact: Poor performance, excessive database load, and slow user experience.

Buggy Code:

import sqlite3

def get_product_details():
    conn = sqlite3.connect('inventory.db')
    cursor = conn.cursor()
    
    # Get all product IDs
    cursor.execute("SELECT id FROM products")
    product_ids = [row[0] for row in cursor.fetchall()]
    
    products_with_categories = []
    # N+1 problem: Executing a new query for each product
    for product_id in product_ids:
        cursor.execute(
            "SELECT category_name FROM categories c "
            "JOIN product_categories pc ON c.id = pc.category_id "
            "WHERE pc.product_id = ?", 
            (product_id,)
        )
        categories = [row[0] for row in cursor.fetchall()]
        
        cursor.execute("SELECT name, price FROM products WHERE id = ?", (product_id,))
        name, price = cursor.fetchone()
        
        products_with_categories.append({
            'id': product_id,
            'name': name,
            'price': price,
            'categories': categories
        })
    
    conn.close()
    return products_with_categories

Fixed Code:

import sqlite3

def get_product_details():
    conn = sqlite3.connect('inventory.db')
    conn.row_factory = sqlite3.Row  # Enable row factory for easy dictionary access
    cursor = conn.cursor()
    
    # Single query with JOINs to fetch all data efficiently
    cursor.execute("""
        SELECT p.id, p.name, p.price, c.category_name
        FROM products p
        LEFT JOIN product_categories pc ON p.id = pc.product_id
        LEFT JOIN categories c ON pc.category_id = c.id
        ORDER BY p.id
    """)
    
    rows = cursor.fetchall()
    conn.close()
    
    # Process the rows to create the product dictionary
    products_dict = {}
    for row in rows:
        product_id = row['id']
        if product_id not in products_dict:
            products_dict[product_id] = {
                'id': product_id,
                'name': row['name'],
                'price': row['price'],
                'categories': []
            }
        
        if row['category_name']:  # Handle products with no categories
            products_dict[product_id]['categories'].append(row['category_name'])
    
    return list(products_dict.values())

Fix Explanation:

The N+1 query problem was fixed by replacing multiple individual queries with a single JOIN query that fetches all necessary data at once. This significantly reduces database round-trips and improves performance. The code also uses SQLite's row factory feature to make working with results more convenient.

Best Practices for Database Operations

  • Always use parameterized queries to prevent SQL injection
  • Properly manage database connections using context managers or try-finally blocks
  • Optimize query patterns to minimize round-trips to the database
  • Use appropriate indexes to speed up queries
  • Implement connection pooling for applications with high concurrency
  • Validate input data before attempting database operations
  • Use database transactions for operations that need to be atomic
  • Handle database-specific exceptions appropriately
